# 백테스트 엔진 설계 (v1: 단일 종목)

## 배경

`docs/enhancement_roadmap.md`가 0순위로 지정한 항목. 이후 제안(ADX 필터, 돌파 버퍼,
Chandelier Exit, 변동성 게이트, 상대강도 랭킹)은 전부 "이 유니버스(KRX+Upbit)에서
유효한지" 검증 수단이 없으면 도입 여부를 판단할 근거가 없다. 이 문서는 그 검증
수단인 백테스트 엔진의 v1 범위를 정의한다.

v1은 **단일 종목** 엔진이다. 멀티종목 포트폴리오 백테스트(자금 배분, 유니버스의
과거 시점 재현/생존편향 처리)는 범위 밖 — 단일 종목 엔진이 검증되고 나서 다음
단계로 확장한다.

## 목표

- 기존 라이브 파이프라인(`turtle/signals.py`, `turtle/stoploss.py`,
  `turtle/trading_params.py`, `turtle/indicators.py`)의 신호/사이징 로직을
  과거 데이터에 그대로 적용해 성과를 측정한다.
- 신호 계산 로직은 **재사용만** 하고 재구현하지 않는다 — 백테스트 결과가
  라이브 스크리닝과 동일한 룰을 반영한다는 신뢰를 보장하기 위함.
- 사람이 실행해서 결과를 읽을 수 있는 CLI + 텍스트 리포트를 낸다.

## 비범위 (v1에서 하지 않는 것)

- 멀티종목/포트폴리오 백테스트, 유니버스 재현, 생존편향 처리
- 복리 리사이징 (자산 증가에 따른 유닛 크기 재조정)
- 슬리피지/수수료 모델
- ADX 필터, 돌파 버퍼, Chandelier Exit 등 로드맵의 이후 항목 (엔진이 이걸
  검증하는 도구가 되지만, 이번 스펙에는 포함하지 않음)

## 아키텍처

새 모듈:
- `turtle/backtest.py` — 시뮬레이션 엔진 (포지션/피라미드/자본 상태 추적)
- `turtle/backtest_report.py` — 거래 로그 → 요약 지표 계산 및 텍스트 포맷
- CLI: `python -m turtle.backtest --ticker <code> --market STOCK|ETF|CRYPTO --start YYYYMMDD --end YYYYMMDD`

핵심 원칙: 매일 `compute_indicators()`, `classify()`, `compute_trading_params()`를
그날까지의 슬라이스(`df.iloc[:i+1]`)로 재호출한다. 데이터는 시작 시 `DataFetcher`로
전체 구간을 한 번에 당겨오고 루프 중 추가 API 호출은 없다. 새로 짜는 코드는
"여러 날에 걸친 포지션 상태 추적"뿐이며 신호/지표 계산 자체는 라이브 파이프라인과
동일 함수를 그대로 호출한다.

## 데이터 흐름

1. CLI가 `ticker`, `market`, `start`, `end` 파싱, `config.yaml` 로드
2. 해당 market의 `DataFetcher`로 `start`보다 충분히 이전(지표 warm-up용 520일
   룩백, 기존 `pipeline.py`의 관례와 동일)부터 `end`까지 OHLCV 조회
3. `start` ~ `end` 구간을 하루씩 순회하며 `run_day()` 호출 (아래 상태 관리 참고)
4. 매 청산 시 거래 기록(`Trade`)을 로그에 추가
5. 종료 후 `backtest_report.py`가 로그 → 요약 지표 계산, 텍스트 리포트 출력

## 포지션/피라미드/스탑 관리

터틀 룰 원칙: 포지션당 스탑은 하나, 최신 유닛 진입가 기준으로 랫칫업.

- **신규 진입**: 무포지션 상태에서 그날 고가가 `high_55`(당일 트리거) 이상이면
  트리거가로 체결, 그 시점 N 고정. 진입 시점 `params.tradable=False`(유닛 수량
  < 최소단위)면 신호를 스킵한다.
- **피라미딩**: 포지션 보유 중, 종가가 `pyramid_1/2/3_price`(최초 진입가
  + 0.5N/1.0N/1.5N) 이상이면 그 가격에 유닛 1개 추가. 최대 유닛 수는
  `config.yaml`의 `account.max_units_per_asset`(현재 4).
- **스탑 갱신**: 유닛 추가 시마다 전체 포지션의 스탑을 `최신 유닛 진입가 - 2N`으로
  갱신 (기존 유닛 포함). N은 최초 진입일 값으로 고정 — 피라미딩 중 재계산하지
  않는다 (`stoploss.py`의 기존 방식과 동일).
- **청산**: 매일 종가 기준으로 `close <= stop_2n` 또는
  `close <= rolling_low(10)`(`turtle/stoploss.py`의 `check_position()`과 동일
  판단 기준)이면 전체 포지션을 그날 종가로 전량 청산.
- 평단가는 유닛별 진입가의 수량 가중평균으로 계산해 손익에 사용.

## 체결가 가정

트리거가 그대로 체결된다고 가정한다 (슬리피지 없음). `high_55` 돌파 시 그 값에,
피라미드 레벨 도달 시 그 값에 체결. 단순하고 낙관적이지만 v1 범위 — 추후
버퍼/슬리피지 모델 추가 가능.

## 자본/사이징

- 초기 자본 = `config.yaml`의 `account.total_value`
- 매 진입/피라미딩 시점마다 `compute_trading_params()`를 그대로 호출해 유닛
  크기를 계산하되, `account.total_value`는 시뮬레이션 시작 시점 값으로 고정
  (복리 리사이징 없음 — v1 단순화)
- 동시 보유 유닛의 자금이 총자산을 초과하는 경우(음수 잔고)는 v1에서 검증하지
  않는다 — 단일 종목 백테스트에서 `risk_pct=0.01`, 최대 4유닛이면 손실 한도가
  총자산의 4% 수준으로 실질적 위험이 낮음. 멀티종목 확장 시 재검토 필요.

## 리포트

- 거래 로그: 진입일/청산일/체결가/유닛수/평단가/손익(금액, %)
- 요약 지표 5개: CAGR, MDD, 승률, Profit Factor, 거래 횟수
- 포맷은 `turtle/report/telegram.py`의 `format_report()` 스타일을 따르는 텍스트,
  콘솔 출력만 (텔레그램 전송 없음)

## 에러 처리

- 조회 구간의 데이터가 지표 warm-up(최소 200거래일, SMA200 기준)에 못 미치면
  경고 후 종료 (기존 `screen_ticker`가 NaN sma_200을 필터 없이 통과시키는 것과
  달리, 백테스트는 초반 구간에 잘못된 신호가 섞이는 걸 막기 위해 warm-up 기간
  자체를 스킵)
- 개별 날짜 처리 중 예외는 전체 백테스트를 중단시킨다 (라이브 파이프라인의
  "종목별 실패가 배치를 막지 않도록" 관용과 다르게, 백테스트는 결과 신뢰성이
  최우선이라 조용히 넘어가면 안 됨)

## 테스트

- `tests/test_backtest.py`: 합성 OHLCV 픽스처(기존 테스트 스타일)로
  - 단순 진입 → 2N 손절 청산 1회
  - 단순 진입 → 10일 저가 청산 1회
  - 피라미딩 2~3유닛 후 청산, 평단가/손익 계산 검증
  - warm-up 부족 구간 처리 검증
- 실데이터 검증: KRX 종목 1개(예: 삼성전자 005930) 장기간 수동 실행, 거래 로그
  육안 확인 (자동화 테스트 아님, 스모크 체크)
