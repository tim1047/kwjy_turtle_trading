# 터틀 트레이딩 진입 후보 종목 스크리닝 시스템 요구사항 스펙

**문서 버전**: v1.0
**작성일**: 2026-07-04
**대상 독자**: 개발 에이전트 (Claude Code 등)

---

## 1. 프로젝트 개요

### 1.1 목적
터틀 트레이딩(Turtle Trading) 시스템 2(55일 돌파 / 20일 저점 청산)에 기반하여, 국내주식·국내상장 ETF·가상자산 중 **매수 진입 조건에 근접했거나 발생한 종목**을 매일 자동 스크리닝하고, 매수 시 필요한 계산값(ATR, 진입가, 손절가, 매수 수량 등)을 산출하는 시스템.

### 1.2 핵심 산출물
1. **일일 종목 유니버스 필터링**: 유동성·히스토리 조건을 만족하는 후보 종목 리스트
2. **일일 지표 계산**: 각 후보 종목의 N(ATR), 55일 최고가, 20일 최저가 등
3. **매수 신호 알림**: 55일 최고가 돌파 종목 리스트 + 매매 파라미터
4. **일일 리포트**: 신호 발생 종목, 근접 종목, 시장 요약

### 1.3 사용자 페르소나
- 개인 트레이더 (계좌 규모 500만 원 ~ 수억 원)
- 매일 장 마감 후 리포트를 받아 다음 거래일 주문 실행

## 2. 대상 시장 및 데이터 소스

### 2.1 대상 자산군

| 자산군 | 대상 | 거래시간 기준 |
|---|---|---|
| 국내 주식 | KOSPI, KOSDAQ 상장 종목 | 09:00~15:30 KST |
| 국내 상장 ETF | 국내 자산 ETF + 미국 추종 ETF (SPY 추종, 나스닥100 추종 등 환헤지/노헤지 포함) | 09:00~15:30 KST |
| 가상자산 | 국내 원화마켓 상장 코인 (업비트/빗썸 기준) | 24시간 365일 |

### 2.2 권장 데이터 소스

| 자산군 | 1차 권장 | 대안 |
|---|---|---|
| 국내 주식/ETF | KRX 정보데이터시스템 (일봉), 한국투자증권 OpenAPI, 키움 OpenAPI | pykrx 라이브러리, 네이버 금융 크롤링 |
| 가상자산 | 업비트 Open API (`/v1/candles/days`), 빗썸 Public API | CoinGecko API (백업) |

**주의**: API 키·비밀번호는 환경변수 또는 `.env` 파일로 관리. 코드에 하드코딩 금지.

### 2.3 데이터 수집 주기
- **국내 주식/ETF**: 매일 장 마감 후 (16:00 KST 이후) 일봉 데이터 수집
- **가상자산**: 매일 09:00 KST (전날 UTC 종가 확정 후) 일봉 데이터 수집

## 4. 일일 지표 계산 요구사항

### 4.1 계산 대상 지표

각 필터 통과 종목에 대해 **매일 장 마감 후** 다음 값을 계산하여 저장.

| 지표 | 정의 | 계산식 |
|---|---|---|
| `close` | 오늘 종가 | 원천 데이터 |
| `high_55` | **직전 55 거래일** 고가 중 최고값 (오늘 제외) | max(High[t-55 : t-1]) |
| `low_20` | **직전 20 거래일** 저가 중 최저값 (오늘 제외) | min(Low[t-20 : t-1]) |
| `high_20` | 직전 20 거래일 고가 중 최고값 | 시스템 1 참고용 |
| `low_10` | 직전 10 거래일 저가 중 최저값 | 시스템 1 참고용 |
| `TR` | True Range | max(H-L, \|H-C_prev\|, \|L-C_prev\|) |
| `ATR_20` (=N) | 20일 ATR | Wilder 방식: N_today = (19×N_yesterday + TR_today) / 20 |
| `ADX_14` | 14일 ADX | 추세 강도 참고용 (표시만) |
| `avg_volume_20` | 20일 평균 거래량 | 유동성 재확인용 |
| `avg_turnover_20` | 20일 평균 거래대금 | 유동성 재확인용 |

**주의사항**:
- `high_55`, `low_20` 계산 시 **반드시 오늘(t) 제외**. 오늘 캔들 포함 시 돌파 판단 불가.
- ATR 초기값 (첫 20일): 단순 산술평균으로 계산 후, 21일차부터 Wilder 공식 적용.
- 가상자산은 UTC 00:00 기준 일봉 사용 (KST 09:00 마감).

### 4.2 매매 파라미터 계산

각 종목에 대해 "만약 오늘 진입한다면"을 가정한 계산값 제공.

| 파라미터 | 정의 | 계산식 |
|---|---|---|
| `entry_trigger` | 매수 트리거 가격 | `high_55` (이 값 돌파 시 매수) |
| `entry_price_assumed` | 가정 진입가 | 시나리오별: (a) 트리거 가격, (b) 다음날 시가 예측치 |
| `stop_loss_price` | 손절가 | `entry_price_assumed - 2 × N` |
| `pyramid_1_price` | 1차 추가매수가 | `entry_price_assumed + 0.5 × N` |
| `pyramid_2_price` | 2차 추가매수가 | `entry_price_assumed + 1.0 × N` |
| `pyramid_3_price` | 3차 추가매수가 (최종) | `entry_price_assumed + 1.5 × N` |
| `unit_size` | 1 유닛 매수 수량 | `floor((account_value × risk_pct) / N)` |
| `unit_notional` | 1 유닛 투입 금액 | `unit_size × entry_price_assumed` |
| `max_position_notional` | 최대 4유닛 총 투입 금액 | 참고용 |
| `max_loss_per_unit` | 유닛당 최대 손실 | `unit_size × 2 × N` |

**계좌 설정값 (설정 파일로 관리)**:
```yaml
account:
  total_value: 100000000  # 계좌 총자산 (원)
  risk_pct: 0.01           # 유닛당 리스크 비율 (1%)
  max_units_per_asset: 4
  max_units_correlated: 6
  max_units_total: 12
```

### 4.3 유닛 계산 시 특수 케이스

- 계산 결과 `unit_size < 1주` (또는 최소 매매 단위 미만): **"매매 불가"** 로 표시
- 가상자산: 소수점 매수 허용 (거래소별 최소 매매 단위 확인)
- ETF: 정수 주 단위, `unit_size`가 0이면 "계좌 규모 부족" 경고

## 5. 신호 감지 및 분류

### 5.1 신호 상태 분류

매일 종목별로 다음 상태 중 하나로 분류.

| 상태 | 조건 | 액션 |
|---|---|---|
| `BREAKOUT_TODAY` | 오늘 고가 ≥ 어제까지의 55일 최고가 (`high_55`) | **매수 진입 검토** |
| `BREAKOUT_CLOSE` | 오늘 종가 ≥ 어제까지의 `high_55` | 다음날 시가 매수 (보수적 룰) |
| `APPROACHING` | 오늘 종가 ≥ `high_55` × 0.98 (2% 이내) | 관찰 리스트 |
| `NEUTRAL` | 위 조건 모두 미해당 | 대기 |
| `BREAKDOWN` | 오늘 저가 ≤ `low_20` | 참고용 (숏 신호 or 롱 청산 근접) |

### 5.2 "직전 돌파 결과" 필터 (선택 기능)

터틀 원본 규칙 중 "직전 55일 돌파 신호가 이익으로 끝났다면 이번 신호 건너뛰기" 로직 구현.

- 각 종목의 **최근 55일 돌파 이력**을 기록 (돌파일, 청산일, 손익 여부)
- 이번 돌파 신호가 발생했을 때, 직전 돌파의 손익 결과를 조회하여 `skip_flag` 표시
- 설정 파일에서 `use_last_loss_filter: true/false` 로 on/off

### 5.3 안전장치 (Failsafe)
- 위 필터로 건너뛴 신호라도, 그 이후 가격이 훨씬 큰 신고가를 갱신하면 재진입 대상으로 표시

## 8. 비기능 요구사항

### 8.1 성능
- 전체 유니버스(약 국내주식 500 + ETF 200 + 크립토 100 = 800종목) **10분 내 처리 완료**
- 데이터 수집 시 API rate limit 준수 (거래소별 제한 확인 후 스로틀링)

### 8.2 안정성
- 데이터 수집 실패 시 **재시도 3회 (exponential backoff)**
- 특정 종목 데이터 오류가 전체 배치를 중단시키지 않도록 종목별 예외 처리

### 8.3 재현성
- 특정 날짜를 인자로 넘겨 **과거 시점 리포트 재생성** 가능해야 함
  - 예: `python main.py --date 2026-06-01`
- 백테스트 확장을 고려하여 순수 함수 기반으로 지표 계산 로직 분리

### 8.4 검증
- 단위 테스트: ATR 계산, 최고/최저값 계산, 유닛 계산은 **알려진 입력-출력**으로 반드시 테스트
- ATR 계산 결과를 TradingView `ATR(20)` 값과 대조 (오차 1% 이내)

### 8.5 확장성
- 신규 자산군 추가 시 데이터 fetcher 어댑터만 추가하면 되도록 인터페이스 분리
- 시스템 1(20일 돌파 / 10일 청산) 병행 지원 옵션

## 9. 개발 시 유의사항 (에이전트에게 전달)

1. **터틀 룰의 미묘한 부분**: `high_55`, `low_20` 계산 시 **오늘 캔들 제외**를 반드시 지킬 것. 이걸 틀리면 신호가 완전히 왜곡됨.
2. **ATR 계산**: Wilder 방식(EMA 기반)이 원본. 단순 SMA는 초기값에만 사용.
3. **유닛 계산**: 항상 **내림(floor)**. 반올림하면 리스크 초과.
4. **가상자산 UTC 처리**: 크립토는 UTC 00:00 = KST 09:00 기준 일봉. 국내 자산과 시간대 다름을 인지.
5. **거래일 계산**: 국내 시장 휴장일(설, 추석, 임시공휴일) 처리 필요. `pykrx.get_calendar_days` 활용.
6. **하드코딩 금지**: 모든 임계값은 `config.yaml`에서 관리.
7. **로깅**: 유니버스 진입/탈락 종목, 신호 발생 종목은 반드시 로그에 남길 것. 나중에 디버깅에 필수.
8. **초기 MVP 스코프**: 슬랙 알림, 시스템 1 병행은 후순위. 국내주식/ETF만 먼저 완성 → 크립토 추가 순으로.
